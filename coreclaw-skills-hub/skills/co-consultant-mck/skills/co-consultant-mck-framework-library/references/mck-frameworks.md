# McKinsey Frameworks — Detailed Reference

Read this reference when applying McKinsey-specific frameworks.

## McKinsey 7S Framework

### Hard 3S (Easier to change)
| Element | Key Questions |
|---------|-------------|
| **Strategy** | What is the strategy? How is competitive advantage sustained? |
| **Structure** | Functional/Divisional/Matrix? Centralized/Decentralized? |
| **Systems** | Performance evaluation? Budget process? Information systems? |

### Soft 4S (Harder to change)
| Element | Key Questions |
|---------|-------------|
| **Shared Values** | Mission/Vision? Behavioral norms? Penetration depth? |
| **Style** | Leadership approach? Decision patterns? Communication? |
| **Staff** | Talent quality? Hiring/development? Motivation? |
| **Skills** | Core competencies? Organizational capabilities? Gaps? |

### 7S Alignment Matrix
Rate each pair as ○ (aligned), △ (partially misaligned), × (misaligned).
Focus remediation on × pairs first, then △.

### Application Rules
1. Start with Shared Values (center of the model) — they influence everything.
2. Soft 4S take longer to change — plan accordingly.
3. Misalignment between Strategy and Structure is the most common problem.
4. All 7 elements must be assessed — partial analysis misses systemic issues.

## Issue Tree

### Structure
```
Core Issue: [Business question — must be a question, not statement]
├── Sub-issue 1 (MECE with siblings)
│   ├── Hypothesis 1a (testable)
│   └── Hypothesis 1b (testable)
├── Sub-issue 2
│   ├── Hypothesis 2a
│   └── Hypothesis 2b
└── Sub-issue 3
    ├── Hypothesis 3a
    └── Hypothesis 3b
```

### Application Rules
1. Core issue MUST be a question, not a statement.
2. Each level must be MECE (no overlap, no gap).
3. Maximum 3-4 levels deep — deeper loses actionability.
4. Apply 80/20: focus on top 20% by impact × testability.
5. Each terminal hypothesis must be testable with available data.

## Pyramid Principle (Minto)

### Structure
```
Conclusion (Key Message — 1 sentence)
├── Argument A (Why So?) → Evidence A-1, A-2 → So What A
├── Argument B (Why So?) → Evidence B-1, B-2 → So What B
└── Argument C (Why So?) → Evidence C-1, C-2 → So What C
```

### Application Rules
1. Conclusion first — always. Reader should know the answer in sentence 1.
2. 3 (±1) supporting arguments — not 2, not 5.
3. Each argument must be independent (MECE).
4. Every level must pass both "So What?" (upward) and "Why So?" (downward).
5. Evidence must be fact-based, not opinion.

## OVA (Overhead Value Analysis)

### Activity Classification
| Category | Definition | Action |
|----------|-----------|--------|
| **Required** | Legally/contractually mandatory | Maintain at minimum cost |
| **Value-Add** | Directly contributes to business value | Optimize |
| **Non-Value-Add** | Can be reduced or eliminated | Eliminate/outsource/automate |

### Application Rules
1. Map ALL indirect activities before classifying.
2. "Required" does not mean "at current cost level".
3. Quantify savings potential for each non-value-add activity.
4. Validate that eliminating an activity doesn't break value-add dependencies.

## SCP Analysis (Structure-Conduct-Performance)

### Structure
| Dimension | Key Questions |
|-----------|-------------|
| **Structure** | Concentration? Entry barriers? Product differentiation? |
| **Conduct** | Pricing behavior? R&D intensity? Advertising? Collusion? |
| **Performance** | Profitability? Efficiency? Innovation rate? |

### Application Rules
1. Start with Structure — it determines the range of possible Conduct.
2. Conduct mediates between Structure and Performance.
3. High concentration + high barriers = attractive for incumbents, unattractive for entrants.
4. Combine with Profit Pool Analysis for a complete industry view.
