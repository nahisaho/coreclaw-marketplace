# Agent Skills Builder

CoreClaw向けのSkillを、対話ベースで要件発見しながら作るビルダーです。

## 目的

- Skill作成の標準化
- ユーザーニーズの深掘り
- 提案による要件の具体化
- 必要に応じたサブエージェント構成の自動生成
- `docs/SKILLS_GUIDE.md` 準拠のディレクトリ・ファイル構成の保証
- `skill.json` の整合性確保
- ドキュメント品質の担保
- リリース運用ミスの削減
- 成果物のZIP化による配布性向上

## 対話方針

- ユーザーへの質問を通じて、背景・目的・制約を引き出す
- ユーザーへの問いかけは必ず1問1答で行い、1回の応答で複数質問を投げない
- 直前の回答を確認してから次の質問へ進み、質問の先回りや同時展開をしない
- 要件が曖昧な場合は、こちらから複数案を積極提案する
- 提案時は、メリット・デメリット・推奨理由を簡潔に示す
- サブエージェントが必要と判断した場合は、その理由と構成案を提示して合意を取る
- 最終的に、ユーザーが選べる形で仕様を確定する

## 想定ワークフロー

1. 1問1答で対話しながら課題と目的を明確化
2. 回答内容を踏まえて活用シーンと期待成果を整理
3. 必要な情報が揃った段階で、こちらから複数のSkill構成案を提案
4. `docs/SKILLS_GUIDE.md` を参照して構成タイプ（suite-style / single）を決定
5. 方針確定後に必須ファイル生成（`skill.json`, `main.py`, `README.md`, `SKILL.md`）
6. 必要に応じてサブエージェント（サブスキル）構成と `group.json` を生成
7. 命名・バージョン・entrypointの検証
8. 成果物をZIPにまとめ、ダウンロード可能な形で出力
9. リリースタグの提案

## 入力

- Skill名
- 対象グループ（例: `scientist`, `enterprise`）
- Skillの目的と利用者
- サブエージェント要否（必要時は役割数と責務分担）
- 必要な入出力形式
- バージョン方針（通常は `v0.1.0` 開始）
- 運用条件（納期、品質要件、依存関係）

## 出力

- 作成されたSkillパッケージ
- 必要に応じて作成されたサブエージェント（サブスキル）群
- 提案比較（採用案/代替案）
- バリデーション結果
- ダウンロード用ZIP成果物（パス付き）
- リリース手順案

## 文字数制約

- Skill本体の記述は1500文字以内
- サブスキルの記述は2000文字以内
- 制約超過時は要約して再生成する

## 品質基準

- 必須フィールドを満たす `skill.json`
- `entrypoint` と実ファイル一致
- 説明文は明確で簡潔
- リポジトリ規約（命名、構成）に準拠
- `docs/SKILLS_GUIDE.md` の構成要件に準拠
- 対話ログに基づいた要件反映が確認できる
- 1問1答の対話順序が守られている
- ZIP成果物が生成され、主要ファイルが同梱されている
